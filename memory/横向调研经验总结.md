# 横向调研经验总结

_基于 2026-04-01 architecture 维度 70 个项目横向调研的实战经验_

---

## 🏗️ 架构概览

### 工作流程（SOP 五阶段）
1. **Phase 0**：准备（目录、项目列表、进度文件）
2. **Phase 1**：信息提取（批量读取纵向报告，生成 info_summary.md）
3. **Phase 2**：初步分类（定义类别→分类→四维评分→候选代表→注册表）
4. **Phase 3**：源码深调（每个项目基于源码生成 deep/scorecard/classification_review）
5. **Phase 3.5**：类别汇总（category_overview + 每类 summary + 五维评分 + 代表确认）
6. **Phase 4**：最终报告（final_report.md）

### 文件结构模板
```
ResearchReports-H/{dimension}/
├── final_report.md
├── .work/
│   ├── progress.md
│   ├── project_list.md
```

## ⚠️ 关键教训（用血泪换来的）

### 1. 子代理策略

**问题**：子代理频繁超时或输出不完整（42秒就停了，只列了目录）
**解决**：
- ⏱️ **超时设为 3600 秒（1小时）**，不要用默认的 300 秒
- 📦 **每个子代理最多处理 3 个项目**，不要贪多
- 🔍 **完成后必须验证文件是否真正写入**，不能只看子代理的返回状态
- 🔄 **异常子代理立即重试**，不要等到最后才发现

### 2. 子代理稳定性问题

**观察**：约 10-15% 的子代理会异常退出（运行 40-50 秒就停，输出不完整）
**模式**：
- 正常完成：运行 7-13 分钟，输出完整
- 异常退出：运行 40-50 秒，只输出一两行
**处理**：检查文件是否存在 → 不存在则立即重试

### 3. 文件路径陷阱

**问题**：子代理可能把文件写到子目录而非指定目录
**案例**：fount/gemini-cli/goose 的文件被写到 `analysis/fount/` 而非 `analysis/`
**预防**：验证时不仅检查文件数，还要检查文件位置

### 4. 不要同时启动多个子代理

**问题**：同时启动 5-6 个子代理导致大量超时
**解决**：一次只启动 1 个子代理，等完成后再启动下一个
**效率**：单子代理 3 个项目约 10 分钟，70 个项目约 4 小时（实际含重试约 5 小时）

### 5. 直接生成 vs 子代理

**大文件适合直接生成**：
- info_summary.md（汇总表）→ 主 Agent 直接写更可靠
- classification_draft.md → 主 Agent 直接写
- classification_registry.md → 主 Agent 直接写

**源码分析必须用子代理**：
- 需要读取大量源码文件
- 需要进入源码目录实际调研
- 主 Agent 上下文有限，容易输出混乱

### 6. 输出混乱问题

**问题**：主 Agent 在生成大文件时出现严重输出混乱（大量乱码、重复文本）
**原因**：上下文过长 + 复杂 PowerShell 脚本 + 多层嵌套
**预防**：
- 大文件内容直接用 write 工具，不要用 exec + PowerShell 拼接
- 避免 exec 中执行复杂的 PowerShell 脚本
- 子代理 task 描述要简洁清晰，不要过于复杂

### 7. 进度文件管理

**问题**：progress.md 被输出混乱污染
**预防**：
- progress.md 只用 edit 工具修改，不用 write 覆盖
- 每个阶段完成后更新进度
- 用 checklist 格式，清晰标注完成状态

## 📊 效率数据

### 子代理性能（Phase 3 源码深调）
- **正常完成率**：约 85%（12 次异常 / 约 23 次调用）
- **平均耗时**：正常 8-13 分钟/3 个项目
- **异常耗时**：40-50 秒就退出
- **总耗时**：约 5 小时（70 个项目）
- **最优批次大小**：3 个项目/子代理

### 子代理 token 消耗
- 正常：50-100K tokens/子代理
- 异常：5-25K tokens/子代理（浪费）

## 🔧 推荐配置

### 子代理 spawn 参数
```typescript
sessions_spawn({
  label: "phase3-batchN",
  mode: "run",
  runTimeoutSeconds: 3600,  // 1小时，不要用 300
  task: "..."  // 简洁明确的任务描述
})
```

### 任务描述模板
```
你是横向调研 Phase 3 的源码深调子代理。请对以下3个项目进行源码级架构分析。

**你的项目**：{project1}, {project2}, {project3}
**维度**：{dimension}

## 路径（明确给出绝对路径）
- SOURCE_CODE_DIR = ...
- ANALYSIS_DIR = ...
- TEMPLATE_DIR = ...

## 源码目录映射（注意大小写）
- {project1} → {absolute_path}

## 初步分类
- {project1} → {category}

## 执行步骤（编号清晰）
1. 读取3个模板
2. 读取纵向报告
3. 进入源码目录调研
4. 生成3个文件（deep/scorecard/classification_review）
5. 验证文件存在且非空
6. 输出交付摘要

## 约束（精简）
- 必须基于源码分析
- [源码分析]/[事实]/[信息不足]标注
- 严格套用模板
```

## 🎯 未来优化方向

1. **自动化验证脚本**：每个 Phase 完成后自动检查文件完整性
2. **子代理心跳检测**：子代理启动 2 分钟后检查是否仍在正常运行
3. **增量式 progress.md**：避免一次性写大文件导致混乱
4. **并行安全数**：确认系统支持的最大并行子代理数（本次发现限制为 5 个）
5. **分类一致性检查**：Phase 3 结束后自动比对 classification_review 中的调整与 registry 是否一致
